git总结

git简介

  • git:分布式版本控制系统

Windows下git的安装与配置

安装完成后的设置

注意git config命令的–global参数，用了这个参数，表示你这台机器上所有的Git仓库都会使用这个配置，当然也可以对某个仓库指定不同的用户名和Email地址。

创建版本库

版本库又名仓库，英⽂名repository，你可以简单理解成一个目录，这个目录里面的所有文件都可以被Git管理起来，每个文件的修改、删除，Git都能跟踪，以便任何时刻都可以追踪历史，或者在将来某个时刻可以“还原”。

创建空目录

Windows系统，为了避免遇到各种莫名其妙的问题，请确保目录名（包括⽗目录）不包含中⽂。

git init

把文件添加到版本库

所有的版本控制系统,其实只能追踪文本文件的改动

二进制文件，虽然也能由版本 控制系统管理，但没法跟踪⽂件的变化，只能把⼆进制文件每次改动串起来，也就是只知道图片从100KB改成了120KB，但到底改了啥，版本控制系统不知道，也没法知道。

不幸的是，Microsoft的Word格式是二进制格式，因此，版本控制系统是没法跟踪Word文件的改动的，前面我们举的例子只是为了演示，如果要真正使用版本控制系统，就要以纯文本方式编写文件。

强烈建议使用标准的UTF-8编码

不要使用Windows记事本!!

建立文件,编写内容,保存为gitts.txt,然后:

  • git add gitts.txt
  • 把文件提交到仓库

-m 后面输入的是本次提交的说明，可以输⼊任意内容，当然最好是有意义的，这样你就能从历史记录里方便地找到改动记录。

1 files changed, 1 insertions(+) 一个文件改动,改动一行

时光机穿梭

  • 运行git status 查看修改前结果
  • 修改文件后,运行git status 查看修改结果

git status命令可以让我们时刻掌握仓库当前的状态，上面的命令告诉我们，readme.txt被修改过了，但还没有准备提交的修改。

  • 使用git diff查看修改内容
    • 减号代表丢失
    • 加号代表增加

总结:

  • 每次都要git add
  • 每次git commit都要加-m解释
  • git status查看状态
  • git diff 查看修改内容

版本回退

  • 使用git log查看修改日志
  • 按q键退出
  • 没做好解释就像上面一样改了啥东西看不懂
  • log最上面的是最新commit的
  • 如果感觉输出信息太多,可以使用:
    • git log –pretty=oneline
  • 使用git reset –hard HEAD^ 回退到上个版本
    • HEAD表示当前版本(注意大写)
    • HEAD^表示上个版本
    • HEAD^^表示上上个版本
    • 以此类推,但是^表示不过来可以写成HEAD~100来表示回退到第100个版本
  • 使用git log查看日志,发现只有 a17c2cc 到之前的记录
  • 如果没有关闭git窗口,想恢复到回退之前的版本,可以
    • $ git reset –hard xxx(xxx代表head指针)
      • 不用写全,能和别的head指针区分就行了

使用git reflog查看历史操作

工作区和暂存区

工作区

工作区（Working Directory）：就是你在电脑里能看到的目录

版本库

版本库（Repository）：工作区有一个隐藏目录“.git”，这个不算工作区，而是Git的版本库。

Git的版本库里存了很多东西，其中最重要的就是称为stage（或者叫index）的暂存区，还有Git为我们⾃动创建的第一个分支master，以及指向master的一个指针叫HEAD。

执行步骤

  1. git add:把文件添加到暂存区
  2. git commit:暂存区所有内容提交到当前分支
    1. 创建版本库时,git自动创建了master分支,所以commit是提交到master分支

管理修改

第一次修改文件后git add,然后第二次修改文件,git commit,只是提交第一次修改的内容到分支上面,第二次修改的内容还没有提交到分支里面

解决:

  • 第二次修改后再次加入暂存区,一起commit
  • commit后再add到暂存区,再次commit

丢弃修改

情景:

  1. 修改后没有放入暂存区
  2. 修改后放入了暂存区(git add …)

没有放入暂存区:

$ git checkout – gitts.txt

  • git checkout –中的”–”很重要,没有”–”,就变成了创建新的分支

放入了暂存区:

  • 从暂存区回退到工作区:
    • git reset HEAD gitts.txt
  • 从工作区回退文件修改内容:
    • git checkout – gitts.txt

删除文件

  • 提交到分支

  • 删除文件(资源管理器删除或者rm 文件名)

  • 查看git状态

现在有两个选择:

  • 从版本库中删除并提交

    $ git rm 文件名

  • 还原文件

    $ git checkout – 文件名

命令git rm赢于删除一个文件。如果一个文件已经被提交到版本库，那么你永远不用担心误删，但是要小心，你只能恢复文件到最新版本，你会丢失最近一次提交后你修改的内容。

远程仓库

创建SSH Key

  • 查看用户文件夹是否存在.ssh目录,如果没有可以创建

    $ ssh-keygen -t rsa -C “youremail@example.com”

  • 创建后有两个文件id_rsa和id_rsa.pub

    • id_rsa是私钥,不能告诉任何人
    • id_rsa.pub是公钥,可以告诉任何人

  • 打开github,在设置中添加你得到的id_rsa.pub内容

添加远程库

  • 创建远程仓库

…or create a new repository on the command line

echo “# test” >> README.md
git init
git add README.md
git commit -m “first commit”
git remote add origin git@github.com:yuhaihua200/test.git
git push -u origin master

…or push an existing repository from the command line

git remote add origin git@github.com:yuhaihua200/test.git
git push -u origin master

  • 创建后有两个提示:
    • 创建一个新的库并推送到github
    • 推送一个存在的库到github
  • 第一次推送git push -u origin master要加参数u
    • 参数u是把本地master分支与远程master分支关联
    • 第二次推送就不再需要加参数u
  • 推送实际上是把本地master分支推送到github
  • 远程库的名字叫origin,也可以更改
  • 第二次及以后的推送

    $ git push origin master

克隆远程库

git clone git@github.com:your_name/responsity_name

可以使用https协议

你也许还注意到，GitHub给出的地址不止一个，还可以用https://github.com/michaelliao/gitskills.git
这样的地址。实际上，Git⽀持多种协议，默认的git://使用ssh，但也可以使用https等其他协议。
使⽤https除了速度慢以外，还有个最⼤的⿇烦是每次推送都必须输⼊⼝令，但是在某些只
开放http端⼝的公司内部就⽆法使⽤ssh协议⽽只能⽤https。

分支管理

  • 一条分支可以形成一条时间线
  • master指向提交
  • HEAD指向当前分支

  • 创建并切换到dev分支

  • git checkout -b表示创建和切换,等于下面两条命令

    • $ git branch dev
    • $ git checkout dev

  • 查看当前分支

    $ git branch

  • 带*号的表示在当前分支
  • 修改dev分支的内容,然后切换回master分支,发现在dev分支修改的内容不见了

  • 合并分支

    $ git merge dev

  • 删除分支

    $ git branch -d dev

总结:

  • Git鼓励大量使用分支：
    • 查看分支：git branch
    • 创建分支：git branch name
    • 切换分支：git checkout name
    • 创建+切换分支： git checkout -b name
    • 合并某分支到当前分支： git merge name
    • 删除分支： git branch -d name
